# QuickTextFormatter Architecture

Last updated: 2026-06-28

## Purpose

QuickTextFormatter is the Grova case converter website at `https://www.quicktextformatter.com`. It provides browser-based text case conversion, text analytics, guides, legal/support pages, multilingual routing, SEO, GA4 analytics, and static sitemap/robots generation.

## Runtime And Deployment

- Frontend hosting: Vercel, deployed from GitHub `main`.
- Frontend framework: Next.js Pages Router in `pages`.
- React stack: React 18, Next.js 16 preview, Tailwind CSS 3.
- Backend boundary: no active backend dependency for the converter workflow. Text conversion and analytics run in the browser.
- SEO discovery files are generated during `prebuild` by `scripts/generate-seo-files.mjs`.
- Vercel Analytics and GA4 load globally from `pages/_app.jsx`.

## Main Directory Map

- `pages`: Next.js pages router routes.
- `pages/index.jsx`: main converter, analytics panel, controls, trust content, and SEO content.
- `components`: navbar, footer, cookie consent, analytics components, SEO head, language switcher.
- `utils/converters.js`: case conversion and line-selection logic.
- `utils/analytics.js`: text analytics logic.
- `utils/i18n.js` and language config files: locale lookup and translated content.
- `utils/site.js`: canonical URLs, alternate links, robots, sitemap, and JSON-LD helpers.
- `middleware.js`: locale redirect middleware.
- `scripts/generate-seo-files.mjs`: build-time sitemap/robots generation.
- `public/sitemap.xml` and `public/robots.txt`: generated static search engine discovery files.

## Route Architecture

Core English routes:

- `/`
- `/about`
- `/contact`
- `/privacy`
- `/terms`
- `/advertising`
- `/guide`
- `/analytics-guide`
- `/tools-directory`
- `/404`

Localized routes are generated by middleware and page logic for:

- `ar`
- `hi`
- `tr`
- `el`
- `es`

The default locale is `en` and stays at the root path. Non-default locales use a leading locale segment, for example `/es`, `/es/about`, and `/ar/privacy`.

## Data Flow

The converter is client-side:

1. The user pastes text or uploads a `.txt` file.
2. The page stores text in React state.
3. `utils/analytics.js` computes counts and text structure.
4. `utils/converters.js` applies the selected conversion options.
5. The browser handles copy and `.txt` download output.

No text processing backend is required for the current architecture.

## Auth, Plans, And Billing

- No Clerk auth is used in this site.
- No subscription or billing portal is currently wired into this site.
- The current product architecture is a public, browser-first utility.

## SEO, Analytics, And Indexing

- Canonical URL and hreflang helpers live in `utils/site.js`.
- Production canonical base comes from `NEXT_PUBLIC_SITE_URL`, with fallback to `https://www.quicktextformatter.com`.
- Localhost and Vercel preview hostnames are rejected as canonical bases.
- `SeoHead` is used on pages to load title, description, canonical, hreflang, Open Graph, Twitter metadata, and JSON-LD.
- Global schema helpers include WebSite, Organization, WebPage, BreadcrumbList, SoftwareApplication, ContactPage, and FAQPage.
- `scripts/generate-seo-files.mjs` writes `public/sitemap.xml` and `public/robots.txt` before builds.
- In Vercel preview environments, robots generation disallows all crawling.
- GA4 is loaded globally through `components/GoogleAnalytics`.
- Custom interaction tracking is loaded through `components/AnalyticsEvents`.

## Operational Notes

- Keep text processing browser-side unless there is a clear reason to introduce backend processing.
- Keep file uploads limited to text files and enforce client-side size limits.
- Keep localized routes, sitemap alternates, and hreflang metadata in sync when adding pages.
- Do not add verification meta tags; Search Console verification is managed outside the codebase.
- When adding a page, update page content, localized copy, `scripts/generate-seo-files.mjs`, sitemap generation, navbar/footer links, and schema metadata together.
